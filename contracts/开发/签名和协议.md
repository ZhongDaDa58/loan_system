#### **3. POST `/api/v1/agreement/sign` - 签署协议**

```json

  "signatureImageUrl": "/signatures/20240505/abc123.png",
  "deviceInfo": "Android 14, Xiaomi 13 Pro"
}
```


**返回数据：**
```json
{
  "code": 200,
  "msg": "协议签署成功",
  "data": null
}

**前端应用场景：**

```

### **二、电子签名接口（SignatureController）**

#### **6. POST `/api/v1/signature/upload` - 上传签名（Base64方式）**

**接口作用：**
- 接收前端传来的Base64编码签名图片
- 保存图片到服务器文件系统
- 计算图片哈希值（防篡改）
- 设置为默认签名
- 返回签名图片的访问URL

**请求参数：**
```
Content-Type: application/json

Body (纯字符串):
"data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..."
```


**返回数据：**
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "id": 1,
    "signatureImageUrl": "/signatures/20240505/uuid123.png",
    "isDefault": 1,
    "createTime": "2024-05-05 10:30:00"
  }
}
```


**前端应用场景：**

📱 **Android端 - 注册流程第3步：采集签名**
```
用户完成协议签署后
  ↓
跳转到"创建您的电子签名"页面
  ↓
展示空白签名板（白色背景，黑色画笔）
  ↓
用户用手指在屏幕上签名
  ↓
提供功能按钮：
[重写] [旋转] [调整粗细]
  ↓
签名完成后点击"确认"
  ↓
前端将签名转为PNG图片 → Base64编码
  ↓
调用此接口上传
  ↓
上传成功后显示：
"✓ 签名创建成功！"
  ↓
自动进入下一步"实名认证"
```


**技术实现要点（Android）：**
```kotlin
// 1. 自定义签名View（继承View类）
class SignatureView : View {
    private val path = Path()
    private val paint = Paint()
    
    override fun onDraw(canvas: Canvas) {
        canvas.drawPath(path, paint)
    }
    
    override fun onTouchEvent(event: MotionEvent): Boolean {
        when (event.action) {
            MotionEvent.ACTION_DOWN -> path.moveTo(event.x, event.y)
            MotionEvent.ACTION_MOVE -> path.lineTo(event.x, event.y)
        }
        invalidate() // 重绘
        return true
    }
    
    // 导出为Bitmap
    fun getSignatureBitmap(): Bitmap {
        val bitmap = Bitmap.createBitmap(width, height, Bitmap.Config.ARGB_8888)
        val canvas = Canvas(bitmap)
        draw(canvas)
        return bitmap
    }
}

// 2. 转为Base64
val bitmap = signatureView.getSignatureBitmap()
val byteArrayOutputStream = ByteArrayOutputStream()
bitmap.compress(Bitmap.CompressFormat.PNG, 100, byteArrayOutputStream)
val base64 = "data:image/png;base64," + 
             Base64.encodeToString(byteArrayOutputStream.toByteArray(), Base64.NO_WRAP)

// 3. 调用接口
api.uploadSignature(base64)
```

## 🎯 **完整用户旅程中的接口调用顺序**

### **场景A：新用户注册流程**

```
1. 用户打开APP → 启动页 → 引导页
   ↓
2. 注册页面：输入手机号、密码、验证码
   ↓
3. 点击"注册" → 后端创建用户账户
   ↓
4. 【调用】GET /api/v1/agreement/list
   获取协议列表，展示给用户
   ↓
5. 用户点击"隐私政策" → 【调用】GET /api/v1/agreement/1
   阅读协议全文
   ↓
6. 用户勾选所有协议
   ↓
7. 【调用】POST /api/v1/signature/upload
   上传手写签名（首次创建）
   ↓
8. 【调用】POST /api/v1/agreement/sign （循环4次）
   签署4个协议
   ↓
9. 【调用】GET /api/v1/agreement/check-all-signed
   验证是否全部签署完成
   ↓
10. 跳转到"实名认证"页面
```


---

```
1. 用户输入手机号、密码登录
   ↓
2. 登录成功，获取Token
   ↓
3. 【调用】GET /api/v1/agreement/check-all-signed
   检查协议签署状态
   ↓
4a. 如果返回 false：
    强制跳转到协议签署页面
    要求补充签署缺失的协议
    
4b. 如果返回 true：
    正常进入首页

## 🌐 **Web审批端是否需要这些接口？**

**答案：基本不需要**

审批端主要关注的是：
- 查看贷款申请详情
- 审批决策（通过/拒绝）
- 查看用户资料

**唯一可能用到的场景：**

🌐 **Web审批端 - 查看用户签署记录（审计用途）**

审批员查看某笔申请的详细信息
  ↓
需要确认用户是否已签署借款合同
  ↓
【调用】GET /api/v1/agreement/my-history?userId=xxx
  查看该用户的所有签署记录
  ↓
确认用户已签署所有必需协议
  ↓
作为审批通过的参考依据
```


但这个功能可以放在后续的"审批员工作台"阶段实现，不是第一阶段的必需功能。

### 第一阶段：基础组件开发（原子功能）
1.  **手写签名板组件 (SignatureView)**
    *   实现一个自定义 View，支持手指滑动绘制轨迹。
    *   实现“清空/重写”功能。
    *   **关键点**：实现将画布内容导出为 `Bitmap` 并转换为 `Base64` 字符串的工具方法。
2.  **协议富文本展示组件**
    *   实现一个能够渲染 HTML 内容的页面（通常使用 `WebView` 或第三方富文本库）。
    *   确保支持缩放和长文本滚动，因为协议内容通常很长。

### 第二阶段：单点接口联调（独立测试）
3.  **签名上传功能**
    *   在一个测试页面调用签名板，获取 Base64 后调用后端 `POST /signature/upload`。
    *   验证是否能成功返回图片 URL，并在页面上显示出来。
4.  **协议列表与详情查看**
    *   调用 `GET /agreement/list`，用列表展示所有协议。
    *   点击列表项跳转到详情页，调用 `GET /agreement/{id}` 并渲染 HTML。
    *   **关键点**：实现“未读到底部不可点击同意”的交互逻辑。

### 第三阶段：核心业务流程串联（注册集成）
5.  **改造注册流程（插入中间页）**
    *   在“手机号注册成功”之后，“进入首页”之前，插入两个强制页面：**签名采集页** -> **协议签署页**。
    *   实现**禁用返回键**逻辑，防止用户跳过这些合规步骤。
6.  **协议批量签署逻辑**
    *   在协议签署页实现多选框逻辑（必须全选才能提交）。
    *   编写循环逻辑：拿到签名 URL 后，依次调用 4 次 `POST /agreement/sign` 接口。
    *   处理并发或顺序执行的错误捕获（如果其中一个失败，如何提示用户）。

### 第四阶段：状态拦截与闭环（完善体验）
7.  **登录后的合规检查**
    *   在 APP 启动或登录成功的回调中，调用 `GET /agreement/check-all-signed`。
    *   根据返回结果判断：如果未完成，强制跳转回“协议签署页”；如果已完成，才允许进入首页。
8.  **默认签名的复用**
    *   在进入“协议签署页”时，先调用 `GET /signature/default`。
    *   如果已有签名，直接显示预览并允许一键签署；如果没有，才引导去画签名。

### 第五阶段：后续优化（非首期必需）
9.  **协议更新弹窗**
    *   根据登录接口返回的版本比对标记，实现“协议已更新”的弹窗提示。
10. **签名管理页面**
    *   在“个人中心”增加入口，允许用户查看历史签名或重新创建签名。

**总结建议：** 先做**签名板**和**WebView**这两个技术难点，再跑通**单个接口**，最后把它们串进**注册流程**里。这样遇到问题容易定位是组件问题还是逻辑问题。
